Mads Kristensen reported: "logs show success but lights don't change." He observed:

## Outcomes

### 1. Driver Behavior Confirmed (Tank)
Tank re-read v0.1.0 driver code and confirmed the scaffold behavior:
- Every command handler (on/off, setLevel, setColor, setEffect) follows pattern: log.info → sendEvent → sendCommand (stub)
- sendCommand() returns early if deviceIp unset or invalid; HubAction block is fully commented out
- **Zero network traffic** leaves the hub as a result of any command
- Mapped every log line in Mads' screenshot to code line numbers (log table in decisions.md)
- Ruled out network-side-effect explanation for "on after save" anomaly (plausible causes: device already on, companion automation, firmware resume on reconnect)

### 2. Local Capture Playbook (Cypher)
Cypher produced a comprehensive Windows-friendly discovery runbook for revealing the Gemstone local API:
- **Recommended sequence:** Port scan (5 min) → curl probe (5 min) → mDNS discovery (10 min) → Wireshark (30 min) → mitmproxy phone-as-proxy (45 min)
- Includes prerequisites, step-by-step instructions, filter expressions, expected outputs, troubleshooting
- If local API exists: Mads captures traffic, sends to team, Tank wires v0.2.0 local endpoints
- If local API doesn't exist (cloud-only firmware): v0.2.0 uses cloud API exclusively

### 3. v0.2.0 Path Analysis (Tank + Cypher)
Tank captured minimum protocol data needed for v0.2.0:
- **Cloud API:** All endpoints documented (auth, device discovery, on/off, state poll, setLevel, setColor); setEffect endpoint unknown (gap to confirm)
- **Local API:** Pending mitmproxy capture; if exists, contains same endpoint set as cloud API
- **Decision:** User now has three concrete paths:
  - **Path A (Cloud):** ~1 day, fully specced, no capture needed
  - **Path B (Local):** 1 day–∞ (depends on capture success), requires mitmproxy
  - **Path C (Hybrid/Probe):** 5-min port scan + curl probe, fall back to Path A if nothing found
