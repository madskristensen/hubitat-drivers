## Summary

**Status:** COMPLETE

Root cause identified: `coerceRawValue()` corrupts string-typed DPs with numeric-looking values. DP 105 (log brightness) and DP 109 (ember brightness) both expect string types per YAML, but `setRawDP` coerces numeric-looking strings to integers.

## Findings

1. **Hypothesis B (type coercion) — CONFIRMED**
   - `coerceRawValue("5")` returns `Integer 5`, not `String "5"`
   - DP 105 YAML type: string; device expects `"1"–"12"` (quoted strings)
   - DP 109 YAML type: string, optional; device expects `"L0"–"L5"` (capital-L prefix strings)
   - `setRawDP` command documentation warns: "whole numbers become integers"
   - Using `setRawDP` for string-typed DPs is invalid test vehicle

2. **DP 105 resolution unconfirmed**
   - `setLogBrightness("12")` dedicated command sends correct string type per YAML
   - Never tested on real hardware; may actually work
   - Mads must run isolated test before concluding DP 105 is read-only

3. **DP 109 resolution unconfirmed**
   - No inbound driver code for DP 109 (no `dp109` attribute)
   - Marked `optional: true` in YAML — may not exist on all firmware
   - Correct value format is `"L1"–"L5"` (non-numeric strings, not coerced)
   - Likely test used wrong value format (`"1"` instead of `"L1"`)
