## Synthesis Update — Pass 3

All coordinator inline probes (~70 payload/header combinations + 20 port alternates) completed without identifying the routing mechanism. Consolidated findings into canonical fingerprint of 11 confirmed facts:

### Canonical Findings (11 Facts)

1. **Single open TCP port:** 80. All alternates closed.
2. **HTTP method restriction:** Only POST and GET accepted. PUT/HEAD/DELETE → `405 Method Not Allowed`.
3. **No Server header** in responses; suggests custom/minimal framework or Node.js with suppression.
4. **OPTIONS times out** — incomplete HTTP/1.1 or intentional blocking.
5. **Content-Type: application/json required** — form-urlencoded/text/plain/empty → `400`.
6. **JSON object structure required** (not array, string, bool, or number).
7. **ALL URL paths return identical 404** ("Invalid route.") — 30+ paths tested.
8. **Routing mechanism unknown** — ~70 payload field names tested; all 404.
9. **Cloud API patterns fail** — shapes from pygemstone cloud client also return 404.
10. **Custom headers ineffective** — X-Action, X-Command, X-Method, etc. no impact.
11. **Response timing consistent** — 120–160 ms for 404; occasional transient timeouts; no "almost-recognized" signal.

### Interpretation

- **Server is functional and parsing.** Validates HTTP method, Content-Type, and JSON syntax.
- **Routing is opaque.** Either (a) vendor-specific field name, (b) behind handshake/pairing step, or (c) requires token from app pairing.
- **Minimal error surface intentional.** Consistent with security-conscious or vendor-locked firmware.

## mitmproxy Capture Runbook

Full, executable capture plan delivered to Mads for phone proxy setup + HTTPS decryption. See `.squad/research/` and handoff section below.

### Minimum Steps (30 min)

1. **Install:** `pip install mitmproxy`
2. **Start mitmproxy on Windows (admin):** `mitmproxy --listen-host 0.0.0.0 --listen-port 8080`
3. **Phone proxy:** Point Wi-Fi to Windows IP (192.168.1.50 example), port 8080
4. **Phone cert:** Download CA from `http://mitm.it` and install
5. **Capture:** Kill app, restart with capture running, tap On/Off
6. **Export:** `mitmdump -r gemstone-app-capture.flow -q --flow-detail 4`

### Cert Pinning Fallback (if mitmproxy fails)

Three tier-2 options documented:
- **Option 1:** ARP spoof + router tcpdump (high yield if HTTP local; 1–2 hours)
- **Option 2:** Android APK decompile with `jadx` (if Android; 30 min, high signal for source reveal)
- **Option 3:** Router-level tcpdump port 80 (assume plain HTTP; 15 min)
