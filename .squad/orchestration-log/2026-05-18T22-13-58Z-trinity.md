## Work Done

1. **Capability Analysis**
   - Recommended: `GarageDoorControl`, `ContactSensor`, `Refresh`, `Initialize`, `Actuator`, `Sensor`
   - Conditional: `HealthCheck` (local only), `Battery` (if API supports)
   - Rejected: `Switch` capability (too dangerous in garage door domain)
   - Attribute design: `door`, `contact`, `lastActivity`, `healthStatus`, `obstructed`

2. **Parent/Child Decision Tree**
   - Single-opener, single-device driver (no parent/child)
   - Light control: child device if API exposes it (capability: `Switch`, `Light`)
   - Cloud multi-door scenario: parent holds auth + discovery, children per door

3. **Three Driver Path Sketches**
   - **Path A (Cloud MyQ):** High risk; Chamberlain blocked Oct 2023; requires token bootstrap; lastActivity-only health monitoring
   - **Path B (ratgdo Local):** Preferred; ESP32 hardware bridge; HTTP polling on 5–10s interval; full HealthCheck capability
   - **Path C (Generic Relay + Contact):** No new driver needed; covered by Hubitat virtual devices

4. **Folder & Packaging Strategy**
   - Separate HPM packages and manifests for cloud vs. local paths
   - Separate README per driver (cloud must warn about API fragility)
   - No shared Groovy code between paths

5. **Garage-Door-Specific Safety Considerations**
   - Log every open/close at INFO unconditionally (audit trail)
   - No auto-close timer in driver (Rule Machine domain)
   - `close()` rate-limiting if door already closing
   - `open()` guard against already-open state
   - Obstruction detection as attribute (not suppressed)
   - Optimistic state vs. confirmed state pattern
   - No `Switch` capability (confirmed hard no)

6. **Open Questions for Mads (Pre-Commitment Gate)**

   - Acceptable API fragility risk?
   - Primary use case?
   - Does your opener have a built-in light?
