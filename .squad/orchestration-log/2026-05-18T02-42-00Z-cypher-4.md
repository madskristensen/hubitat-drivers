### Item 2: Tuya Autodiscovery on Hubitat Feasibility

**Deliverable:** `.squad/decisions/inbox/cypher-tuya-autodiscovery-feasibility.md`

**Verdict:** ⚠️ **Feasible-with-caveats — primary approach blocked, Plan B viable.**

**Key Finding — Hubitat UDP Limitation:**
- ❌ **Passive UDP broadcast listening:** Not supported by Hubitat (no API for binding to UDP ports)
- Hubitat staff confirmed (2018, unchanged): "You can only send out UDP messages and receive a reply to that message"
- Tuya autodiscovery requires passive listening on port 6666/6667 — **not feasible on Hubitat**

**Plan B — Active TCP Probe (Feasible):**
- ✅ Explicit "Discover" button that performs active TCP scan of /24 subnet on port 6668
- ✅ Solves "DHCP renewal silently breaks the driver" problem
- ✅ ~1 hour of Tank work to implement

**Implementation Flow Documented:**
- Sequential rawSocket connects (no parallel — single-thread Hubitat sandbox)
- DP_QUERY frame sent to each IP; match on gwId to find device
- Smart ±20 range first (typical DHCP drift), then full sweep if needed
- Heartbeat suspended during scan to free socket

**Fallback (Recommended First Step):**
- Document DHCP reservation guidance in README
- Users set static reservation in router — prevents IP changes entirely
- Zero driver code needed

**Risks Identified for Switch:**
- rawSocket throughput: ~2s/IP × 254 IPs = 8 min worst case
- Hub sandbox rate-limiting on rapid sequential TCP connects
- gwId in v3.3 DP_QUERY response must be confirmed on real hardware
- IP outside /24 subnet edge case

**Tuya Protocol Details Captured:**
- Broadcast ports 6666 (v3.1, plaintext) vs 6667 (v3.3, XOR-encrypted)
- v3.3 decryption key documented: `yGAdlopoPVldABfn` (16 bytes, repeated)
- Payload schema for v3.3 broadcasts: `gwId`, `ip`, `productKey`, etc.
