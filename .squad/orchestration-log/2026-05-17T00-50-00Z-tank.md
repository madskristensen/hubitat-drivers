## v0.2.5: Hubitat HTTP Encoder Quirk Fix

### Discovery
v0.2.4 diagnostic logging revealed the true root cause: **Hubitat's `asynchttpPost` method has no encoder for `application/x-amz-json-1.1` content type**. When the params map specified `contentType: application/x-amz-json-1.1`, Hubitat rejected the request locally with error:
```
No encoder found for request content type application/x-amz-json-1.1
```

The synthesized `status=408` / `hasError=true` was not an AWS timeout — it was Hubitat blocking the request before it left the hub.

### Root Cause Analysis
- Cognito requires the **wire header** `Content-Type: application/x-amz-json-1.1` to be sent to AWS.
- Hubitat's HTTP client requires a **known encoder** (`contentType` param) to serialize the body — it does not recognize `application/x-amz-json-1.1` as a valid encoder type.
- Solution: Use `contentType: application/json` to select Hubitat's JSON encoder, while still sending wire header `Content-Type: application/x-amz-json-1.1` via the `headers` map.

### Changes Applied
- Switched Cognito builder `contentType` hint from `application/x-amz-json-1.1` to `application/json`.
- Added `requestContentType: application/json` for clarity.
- Pre-serialize all JSON bodies with `JsonOutput.toJson(...)` before passing as String `body` param.
- Applied same pattern to Gemstone API Gateway calls for consistency.
- Kept wire header `Content-Type: application/x-amz-json-1.1` in `headers` map for Cognito (required by AWS).
- Kept wire header `Content-Type: application/json` in `headers` map for Gemstone API (required by AWS API Gateway).
- Added clearer error message for the `No encoder found ...` case to alert future maintainers that Hubitat is rejecting the request locally, not remotely.
